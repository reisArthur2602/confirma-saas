API do Confirma

Infraestrutura de confirmação de agenda para clínicas. A API é REST sobre HTTPS, com corpos em JSON. Toda a integração gira em torno de um único POST autenticado — o Confirma cuida de fila, agendamento, envio, interpretação da resposta e do callback de volta ao seu sistema.

BASE URL https://api.confirma.dev

Dois ambientes isolados: Sandbox (chaves pk_test_…, envio simulado sem custo) e Produção (pk_live_…). A chave usada define o ambiente.

Quickstart

Da chave à primeira confirmação em menos de 30 minutos. Um único POST autenticado — você delega fila, retry, templates e callback ao Confirma.

{{ d.n }}
Gere sua API key
No painel, em API Keys, copie a chave publicável (pk_test_…) e o secret HMAC. O secret nunca sai do servidor.
Assine o payload (HMAC-SHA256)
Calcule X-Signature sobre o corpo exato da requisição, com o seu secret.
const body = JSON.stringify(appointment);
const sig  = createHmac("sha256", secret)
               .update(body).digest("hex");
Envie o agendamento
Um 202 Accepted confirma o enfileiramento. Não é enviado na hora — dispara no offset.
{{ codeSend }}
Receba o callback
Quando o paciente responde, o Confirma faz POST assinado na sua callbackUrl. Valide a assinatura e atualize a agenda.
app.post("/webhooks/confirma", (req, res) => {
  const sig = createHmac("sha256", secret)
                .update(req.rawBody).digest("hex");
  if (sig !== req.headers["x-confirma-signature"])
    return res.sendStatus(403);
  updateAgenda(req.body); // confirmed | cancelled | ...
  res.sendStatus(200);
});
Teste ponta a ponta
Rode o fluxo no Sandbox — envio simulado, sem custo de mensagem, e veja o callback disparar.

Autenticação

Toda requisição carrega dois cabeçalhos. A chave pública identifica a conta; a assinatura HMAC prova que o corpo não foi adulterado.

HeaderDescrição
X-Api-KeyChave publicável da conta. Enviada no header, nunca no corpo.
X-SignatureHMAC-SHA256 do corpo exato da requisição, com o secret da conta (hex).
NODE.JS
import { createHmac } from "crypto";

const body = JSON.stringify(appointment);
const sig  = createHmac("sha256", secret)
               .update(body).digest("hex");
// envie sig em X-Signature
O secret só existe no seu servidor. Nunca o exponha em cliente, app mobile ou logs.

Idempotência

Envie o header Idempotency-Key (um UUID por agendamento) para garantir que reenvios não gerem notificação duplicada. Uma chave já processada retorna 200 OK com o mesmo appointmentId.

Sem o header, o Confirma deriva a chave de externalId + datetime.

POST /v1/appointments

Callbacks

Em cada mudança de estado relevante, o Confirma faz um POST assinado na sua callbackUrl. Valide o header X-Confirma-Signature e responda 2xx — caso contrário aplicamos retry com backoff.

Templates

Templates são configuráveis por conta e mantidos estritamente transacionais (categoria utility). Variáveis são interpoladas do payload: {{ vPatient }}, {{ vType }}, {{ vDatetime }}.

Uma frase promocional dentro de um template transacional pode causar reclassificação para marketing pela Meta, multiplicando o custo por mensagem.

Rate limits

A rota pública aplica rate limiting por conta para conter abuso. Ao exceder, a API retorna 429 Too Many Requests com o header Retry-After. O limite padrão é 120 req/min, ajustável por conta.

Erros

Erros usam códigos HTTP padrão. O corpo traz code e message; em 422, também fields com os campos inválidos.

{{ er.code }} {{ er.slug }} {{ er.desc }}
